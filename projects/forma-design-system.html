Forma Design System

Building a shared foundation for 6+ product teams — a multi-brand, token-based design system that unified voice, motion, and component patterns across platforms.

01 — Problem

Six teams. Six interpretations of the same button.

Before Forma, each product team had evolved its own component library. What started as pragmatic independence had accumulated into a significant consistency debt — buttons with slightly different border-radii, colour tokens that diverged across brands, and a motion language nobody had formally defined.

Every design handoff involved re-explaining decisions. Engineers rebuilt components that already existed elsewhere. Accessibility annotations were inconsistent or missing entirely. New brand themes required weeks of manual reskinning.

"We had the same component built four different ways in four different repositories. Nobody owned it, so nobody maintained it properly." — Engineering lead

02 — Approach

Start with tokens. Let components follow.

Rather than jumping straight to a component library, we started with a token audit. We mapped every colour, spacing, radius, shadow, and typographic decision in use across all six products. Patterns emerged quickly — and so did the conflicts.

The token architecture came first because without a stable foundation, any component built would inevitably fracture under a new brand theme or dark mode requirement.

03 — Token Architecture

Three tiers. One source of truth.

The Forma token model has three distinct layers. Primitive tokens hold raw values — a specific hex colour or a pixel measurement with no semantic meaning attached. Semantic tokens reference primitives and carry intent: color.interactive.default tells you what the value means, not what it is. Component tokens map semantic tokens to specific component contexts.

This separation meant that switching from Brand A to Brand B required only updating which primitive each semantic token pointed to. Components never needed to change.

Token architecture diagram

Naming convention: [category].[variant].[state] — e.g. color.interactive.hover, space.component.padding-sm. Predictable, readable, grep-able.

04 — Components

Built once. Accessible by default.

Every component was built with a strict accessibility contract: keyboard interaction model documented before code was written, ARIA patterns reviewed against WCAG 2.1, and focus management tested with VoiceOver and NVDA. Components that failed any accessibility criterion were not shipped.

The Figma component library mirrored the code — every prop in code had a corresponding variant in Figma. Annotations directly on components described the accessibility intent so designers could hand off confidently.

05 — Documentation

Documentation that answers the next question.

Most design system documentation answers "how do I use this?". Forma's documentation also answers "why does it work this way?" and "what should I do when this component isn't quite right for my use case?".

Every component page includes: usage guidance, accessibility notes, keyboard interaction table, do/don't examples, and a link to the Storybook story. The documentation became the primary onboarding surface for new designers and engineers joining the organisation.

06 — Outcomes

A shared language changes how teams work.

Eighteen months after launch, Forma is used across all six product teams with no parallel component libraries remaining. Design handoff time dropped significantly as engineers had a stable, documented reference. New brand themes that previously took weeks to implement could be delivered in days by updating semantic token mappings.
